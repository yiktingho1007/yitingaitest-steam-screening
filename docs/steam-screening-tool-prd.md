# Steam 产品初筛工具 PRD

- 状态：Draft v0.2
- 最后更新：2026-05-14
- 面向用户：莉莉丝预研团队
- 文档目标：定义一个接入大语言模型的内部网页工具，用于对 Steam 产品做快速、可追溯、可比较的初步筛选

## 1. 产品概述

### 1.1 项目背景

莉莉丝预研人员在做新品扫描时，往往需要先判断一个 Steam 产品是否值得继续深入研究。现阶段，这个动作通常依赖人工搜索、人工摘录和经验判断，效率不高，而且不同研究人员的判断口径不一致。

因此需要一个内部网页工具，支持输入任意 Steam 游戏后，快速返回：

1. 该产品的核心数据快照
2. 两款竞品的自动对比
3. 一段受控、基于数据的大模型初筛结论

### 1.2 核心目标

构建一个网页工具，能够：

1. 支持通过游戏名、Steam 链接或 `AppID` 输入目标产品
2. 从有效且可追溯的数据源获取 Steam 核心信息
3. 自动挑选两款可比竞品
4. 用 LLM 基于结构化数据生成简洁的筛选结论
5. 明确展示每个关键字段的数据来源、更新时间和可信等级

### 1.3 产品原则

本工具遵循一个核心原则：

**先有可靠数据，再做模型解释。**

也就是说，大模型只负责“解释数据”，不负责“找数据”，更不允许在缺失字段上主观补数。

## 2. 目标用户与使用场景

### 2.1 目标用户

- 预研分析师
- 战略研究人员
- 市场扫描人员
- 需要做早期竞品判断的制作人或产品负责人

### 2.2 主要使用场景

- 快速判断某个 Steam 产品是否值得继续深挖
- 在立项前做一轮轻量产品扫描
- 对一个目标产品和两款竞品做第一轮横向比较
- 在正式研究报告前，产出一版内部预判摘要

## 3. 成功标准

### 3.1 产品成功标准

- 用户能在 10 分钟内完成一次 Steam 产品初筛
- 用户能清楚知道页面上每个关键数字来自哪里
- 用户能把结果直接用于内部讨论，而不是再次从零整理

### 3.2 V1 运行目标

- 缓存命中的情况下，页面结果 5 到 10 秒内返回
- 输入到正确 `AppID` 的识别成功率不低于 90%
- 在 30 到 50 个样本游戏上，竞品推荐的“可用率”不低于 80%
- LLM 输出中不出现无数据支撑的数值结论

## 4. 范围定义

### 4.1 V1 范围内

- 输入解析：游戏名、Steam URL、`AppID`
- Steam 产品匹配与校验
- 核心数据快照展示
- 两款竞品自动对比
- 基于结构化数据的 LLM 结论
- 带来源标注的第三方估算数据接入（如 SteamSpy，可选）
- 数据来源标记
- 数据更新时间展示
- 项目改动记录与执行日志

### 4.2 V1 暂不纳入

- 自研销量预测模型
- 自研收入预测模型
- 自研愿望单预测模型
- 自研留存预测模型
- 多平台商店对比
- 全量市场看板
- 自动化幻灯片输出
- 多语言报告系统

## 5. 数据源策略

这个工具不能把所有数据都混在一起展示，而是要从一开始就做“可信度分层”。

| 等级 | 类型 | 示例来源 | 使用方式 | 规则 |
| --- | --- | --- | --- | --- |
| A 级 | Steam 官方公开且文档明确支持的数据 | `IStoreService/GetAppList`、`ISteamUserStats/GetNumberOfCurrentPlayers`、Steam Reviews API | 作为核心判断依据 | 可以进入比较逻辑和 LLM 结论主链路 |
| B 级 | 高可信补充数据 | Steam 商店公开字段、SteamDB、公开商店详情字段 | 作为补充说明与交叉验证 | 必须标成补充数据，不可单独作为硬结论依据 |
| C 级 | 第三方估算数据 | SteamSpy、VG Insights、Gamalytic、外部峰值趋势站点 | 可作为估算性判断输入 | 可以进入主结论，但必须显式标注来源、估算属性和不确定性 |

### 5.1 V1 推荐纳入的 A 级字段

- `AppID`
- 游戏名称
- 商店可见状态
- 当前在线人数
- 评测总结
- 好评数
- 差评数
- 总评测数
- 评论样本或评论摘要
- 最近拉取时间

### 5.2 V1 允许作为 B 级补充字段

- 价格
- 折扣状态
- 发售时间
- 开发商
- 发行商
- 支持语言
- 标签或类型描述
- 是否处于抢先体验
- 简介

### 5.3 具体数据源建议清单

为避免后续开发阶段对“可信数据源”理解不一致，V1 建议明确区分以下几类来源：

#### 5.3.1 A 级主数据源

- Steam 官方 `IStoreService/GetAppList`
- Steam 官方 `ISteamUserStats/GetNumberOfCurrentPlayers`
- Steam 官方 Reviews API：`store.steampowered.com/appreviews/<appid>?json=1`

这些字段应作为：

- 目标产品核心指标
- 竞品打分主输入
- LLM 主结论依据

#### 5.3.2 B 级高可信补充源

- Steam 商店公开页面字段
- SteamDB

其中，**SteamDB 可以视为比较可信的高质量补充源**，原因包括：

- 它长期聚焦 Steam 数据追踪
- 对应用、包体、价格、更新、分支等信息覆盖较全
- 许多信息本质上仍来自 Steam 自身公开网络或公开页面
- 对预研人员而言，SteamDB 的可读性和历史追踪能力很强

但同时需要注意：

- SteamDB 不是 Valve 官方服务
- SteamDB FAQ 明确说明其本身**不提供公开 API**
- SteamDB FAQ 明确说明**不允许自动抓取或爬取**

因此在本工具中的建议角色是：

- 作为人工复核入口
- 作为交叉验证来源
- 作为历史变更与补充信息参考
- 作为页面中的“查看外部详情”跳转目标

不建议在 V1 中把 SteamDB 作为：

- 主拉数接口
- 高频自动抓取来源
- 唯一真实来源

#### 5.3.3 C 级第三方估算源

- SteamSpy
- VG Insights
- Gamalytic
- 其他销量/拥有量估算服务

这些来源在研究上有明确参考价值，可以进入主结论，但需要遵守以下规则：

1. 必须在页面和 LLM 文本中明确标出来源名
2. 必须明确写成“估算”“推测”或“第三方测算”
3. 不得把估算值表述成 Steam 官方确认数据
4. 如果估算源之间冲突，优先展示区间或保守表述
5. 如果估算值参与竞品判断，应在页面上明确说明“包含第三方估算信号”

### 5.4 数据治理规则

- 每个字段都必须记录 `source`、`fetched_at`、`confidence_tier`
- 关键数字必须在前端显示来源
- 缺失字段必须按缺失显示，不允许推断补全
- LLM Prompt 必须明确禁止编造缺失数据
- C 级字段进入 LLM 结论时，必须在文本中保留来源标识和“估算”属性

## 6. 核心用户流程

1. 用户输入游戏名、Steam 链接或 `AppID`
2. 系统解析并确认目标产品
3. 系统拉取并规范化目标产品数据
4. 竞品引擎生成 5 个候选，并自动选出 2 个最佳竞品
5. 页面展示目标产品和两款竞品的横向对比
6. LLM 接收结构化 JSON 数据
7. LLM 输出：
   - 一句话产品定位
   - 三条与竞品的关键差异
   - 一条是否值得进一步研究的建议
   - 一条数据风险提示
8. 用户据此决定是否继续进入深度研究

## 7. 功能需求

### 7.1 输入与产品匹配

- 支持游戏名、Steam URL、`AppID`
- 对无效输入给出明确提示
- 对重名产品给出候选选择
- 最终落到一个确定的 `AppID`

### 7.2 数据获取与规范化

- 将所有来源统一映射到内部字段结构
- 对应用列表做缓存和增量刷新
- 对当前在线人数做短缓存
- 目标产品拉取与竞品拉取分开，便于重试和降级
- 必要时保留原始响应快照，便于后续排查

### 7.3 竞品推荐

V1 的竞品推荐应采用规则引擎，而不是纯 LLM 主导。

最低评分维度建议包括：

- 标签或类型重叠度
- 价格带相似度
- 发售阶段相似度
- 评论量级相似度
- 当前在线量级相似度

输出要求：

- 先生成 5 个候选
- 自动选择前 2 个
- 后续版本可加入人工替换

### 7.4 LLM 分析层

LLM 只能消费系统传入的结构化字段。

固定输出块：

- 产品定位总结
- 与两款竞品的差异总结
- 是否值得继续研究的建议
- 一条数据边界提示

Prompt 约束：

- 不允许输出无支撑的销量、收入、留存
- 不允许输出未提供的趋势判断
- 对缺失字段必须明确说“暂无可靠公开数据”
- 如果使用 C 级估算字段，必须明确写成“基于 SteamSpy 估算”或“第三方估算显示”
- 语气要简洁、偏决策支持

### 7.5 前端展示

页面应至少包含：

- 搜索输入区
- 目标产品快照区
- 三列对比区
- LLM 总结区
- 数据来源与更新时间标记

展示规则：

- 每个关键数字必须可追溯
- 数据等级必须一眼可见
- 某个数据源失败时，页面仍能局部展示

### 7.6 日志与可审计性

- 记录每次分析请求
- 记录最终命中的 `AppID`
- 记录竞品推荐结果
- 记录数据拉取成功或失败
- 记录模型输出元信息

## 8. 非功能需求

### 8.1 性能

- 缓存命中时尽量控制在 5 到 10 秒
- 未命中时允许变慢，但必须明确展示加载状态

### 8.2 稳定性

- 单一数据源失败不能让整页崩掉
- 页面需要有明确的降级和失败提示

### 8.3 安全性

- 所有 API Key 只保存在后端
- 不允许在前端暴露 Steam Key 或模型 Key
- 支持内部访问控制

### 8.4 可观测性

- 记录响应耗时
- 记录数据源失败率
- 记录缓存命中率
- 记录单次模型成本

## 9. 输出定义

### 9.1 产品快照

目标产品卡片建议展示：

- 名称
- `AppID`
- 评测摘要
- 评论量
- 当前在线
- 关键补充字段
- 来源与更新时间

### 9.2 竞品对比

对比区采用三列结构：

- 目标产品
- 竞品 A
- 竞品 B

每一行展示一个统一字段。

### 9.3 LLM 结果

总结区固定输出：

- 一句话定位
- 三条差异点
- 一条建议
- 一条风险提示

## 10. 系统架构建议

### 10.1 建议模块

- 前端 Web 页面
- 后端 API 服务
- Steam 数据适配层
- 竞品评分服务
- LLM 分析服务
- 缓存与持久化层
- 日志与监控层

### 10.2 建议处理顺序

1. 解析输入并识别产品
2. 拉取结构化数据
3. 做字段标准化与来源标记
4. 搜索竞品候选
5. 评分并选择两款竞品
6. 先渲染数值结果
7. 再调用 LLM 生成结论
8. 写入日志

## 13. 验收标准

V1 满足以下条件时可认为通过验收：

- 用户能通过至少一种支持的输入方式定位目标 Steam 游戏
- 系统能给出带来源标记的目标产品核心数据
- 系统能展示目标产品与两款竞品的统一对比
- LLM 输出不包含无数据支撑的结论
- 部分数据失败时，页面仍能可理解地展示结果
- 项目全程有持续记录的文档轨迹

## 14. 风险与应对

| 风险 | 影响 | 应对方式 |
| --- | --- | --- |
| Steam 公开字段文档支持不一致 | 影响研究信任度 | 强制做 A/B/C 分层，并在 UI 明确标注 |
| 竞品推荐看起来不合理 | 用户不信任结果 | 优先用可解释规则打分，而不是黑盒推荐 |
| LLM 过度发挥 | 初筛结论误导 | 用结构化输入和严格 Prompt 限制 |
| Steam 数据波动或接口失败 | 页面结果不稳定 | 做缓存、重试和局部降级 |
| 范围过早膨胀成完整 BI 产品 | 拖慢交付节奏 | 坚持 V1 只做“初筛” |
